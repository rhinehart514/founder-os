# Roadmap: founder-os

## Overview

v2.0 repositions founder-os from a measurement-first tool to a demand-side product thinking tool. The work proceeds in dependency order: canonical anchors before skill content, skill content before narrative, narrative before advanced features. Each phase delivers something coherent and independently verifiable — the product is demonstrably more demand-first after each phase completes.

## Phases

**Phase Numbering:**
- Integer phases (1, 2, 3): Planned milestone work
- Decimal phases (2.1, 2.2): Urgent insertions (marked with INSERTED)

Decimal phases appear between their surrounding integers in numeric order.

- [ ] **Phase 1: Foundation** - Canonical demand-first config and routing disambiguation
- [ ] **Phase 2: Demand Skills** - Strengthen demand-side skills and integrate /blueprint
- [ ] **Phase 3: Narrative** - Rewrite README, quick-start, and /onboard to lead with demand-thinking
- [ ] **Phase 4: Measurement Reframing** - Reframe /score, /eval, /taste, /retro as demand verification
- [ ] **Phase 5: Advanced Demand** - Four forces mapping and hire/fire switch analysis

## Phase Details

### Phase 1: Foundation
**Goal**: Canonical sources say demand-first — every skill that reads config/founder.yml and mind/ files gets the right anchor
**Depends on**: Nothing (first phase)
**Requirements**: DEMAND-01, DEMAND-02, DEMAND-03, DEMAND-04
**Success Criteria** (what must be TRUE):
  1. config/founder.yml value hypothesis leads with demand-side thinking, not "measurably better every session"
  2. mind/identity.md, mind/standards.md, and mind/thinking.md describe demand-side as the primary job
  3. Each of the 5 overlapping demand skills (/discover, /product, /strategy, /ideate, /research) has a documented non-overlapping job and a clear "use this when" trigger
  4. The outcome hierarchy (outcome → opportunity → system → feature) is surfaced as a user-facing concept, not buried in /discover references
**Plans**: TBD

### Phase 2: Demand Skills
**Goal**: Demand-side skills produce outputs that could drive go/kill decisions — three-dimension job statements, evidence labeling, and /blueprint integrated
**Depends on**: Phase 1
**Requirements**: DEMAND-05, DEMAND-06, DEMAND-07
**Success Criteria** (what must be TRUE):
  1. /blueprint skill is installed and routed within founder-os (not just available globally)
  2. /discover and /blueprint produce explicit three-dimension job statements (functional/emotional/social)
  3. All demand-side skill outputs label every claim by evidence class: verified external (URL), vault-backed, or LLM synthesis
**Plans**: TBD

### Phase 3: Narrative
**Goal**: A new user opening README or running /onboard encounters demand-side thinking before any build or measurement commands
**Depends on**: Phase 2
**Requirements**: NARR-01, NARR-02, NARR-03, NARR-04
**Success Criteria** (what must be TRUE):
  1. README opens with demand-side thesis (JTBD, value props, feature architecture) — /score and /go appear in a later section
  2. Quick-start shows /discover → /blueprint → /decide flow before mentioning /go or /score
  3. /onboard introduces demand-side thinking before the measurement pipeline
  4. All modified skills comply with Anthropic best practices: under 500 lines, progressive disclosure, trigger-inclusive descriptions
**Plans**: TBD
**UI hint**: yes

### Phase 4: Measurement Reframing
**Goal**: Measurement skills describe themselves as verification of demand decisions, not as the product's primary value
**Depends on**: Phase 3
**Requirements**: MEAS-01, MEAS-02, MEAS-03
**Success Criteria** (what must be TRUE):
  1. /score, /eval, and /taste descriptions frame scoring as "did we validate the demand decision we built toward?"
  2. Running /go requires passing a demand check — kill conditions are evaluated before build starts, not after
  3. /retro frames the session as "did the JTBD thesis hold up?" with demand-side learning prompts
**Plans**: TBD

### Phase 5: Advanced Demand
**Goal**: Founders can run four forces analysis and hire/fire switch analysis as named demand-side tools
**Depends on**: Phase 4
**Requirements**: ADV-01, ADV-02
**Success Criteria** (what must be TRUE):
  1. A founder can invoke four forces mapping (push/pull/anxiety/inertia) and receive a structured output naming each force for their product
  2. A founder can run hire/fire switch analysis and receive an output that names the real competitor (often inertia, not another product)
  3. Both tools produce outputs that connect to job statements from Phase 2 — they are not standalone diagnostics
